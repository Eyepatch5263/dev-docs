## Introduction

Before diving into complex, distributed database sharding or consensus algorithms, you must master the fundamental building blocks of modern computer systems. These concepts form the vocabulary of system architecture.

---

## 1. Client-Server Architecture

At the heart of almost every system is the **Client-Server model**. It is a distributed application structure that partitions tasks or workloads between the providers of a resource or service (servers) and service requesters (clients).

<Mermaid chart='flowchart LR[nl]
    Client1["Laptop Client"] -->|Request over HTTP/TCP| LB["Server Gateway"][nl]
    Client2["Mobile Client"] -->|Request over HTTP/TCP| LB[nl]
    LB -->|Response (JSON/Data)| Client1[nl]
    LB -->|Response (JSON/Data)| Client2' />



*   **Client**: The user-facing application (e.g., web browser, mobile app, IoT device) that initiates requests for resources or actions.
*   **Server**: A background computer or application instance that listens for incoming client requests, executes logic, queries databases, and returns the requested resources.

### Stateless Systems
A service is **stateless** if it does not retain any client information (state) on its server between requests. Each request from the client must contain all the information necessary to service it (e.g., JWT auth tokens).
*   **Pros**: Extremely easy to scale. If a server dies, we can spin up a new one immediately. Any server can handle any request.
*   **Cons**: Clients must send authorization and metadata with every request, increasing payload size.

### Stateful Systems
A service is **stateful** if it stores client state (like active user sessions or local files) on its local memory or disk.
*   **Pros**: Convenient for maintaining real-time context (e.g., multiplayer gaming, open TCP sockets).
*   **Cons**: Hard to scale. If Server A crashes, the user's session is lost. You must route the user's subsequent requests to the *exact same server* (sticky sessions).

## 3. Communication & API Protocols

Servers and clients speak to each other using structured APIs. The choice of protocol affects performance, payload size, and ease of development:

| Protocol | Data Format | Architectural Style | Pros | Cons | Best For |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **REST** | JSON, XML | Resource-oriented over HTTP | Human-readable, simple, widely supported. | Over-fetching or under-fetching data. | Public APIs, CRUD apps. |
| **GraphQL**| JSON | Query-driven over HTTP | Client requests exactly what it needs in a single roundtrip. | Complex caching, query parsing overhead. | Mobile client integration. |
| **gRPC** | Protocol Buffers (Binary) | Remote Procedure Call over HTTP/2 | High performance, typed interface, bi-directional streaming. | Hard to inspect (binary format), poor browser support. | Inter-service microservices communication. |

How you organize your codebase and deployment boundaries determines your operational complexity.

### Monolithic Architecture
A monolith puts all application components (auth, billing, catalog, UI) into a single, unified codebase and deployable unit.
*   **When to use**: Early-stage startups, simple applications, small teams.
*   **Pros**: Simple deployment, fast local debugging, zero network latency between modules.
*   **Cons**: Single point of failure, scaling one feature requires scaling the whole application, long build times.

### Microservices Architecture
Microservices break the application into independent, self-contained services that communicate via lightweight network protocols (HTTP/gRPC/Kafka). Each service typically owns its own database.
*   **When to use**: Large engineering organizations, complex platforms with high-scale demands.
*   **Pros**: Independent deployments, choose the best tech stack for each job, faults are isolated to individual services.
*   **Cons**: Network latency, complex distributed debugging, data consistency challenges (Sagas/Two-Phase Commit).

## Key Terms Checklist

*   **API Gateway**: A reverse proxy that routes client requests to their appropriate backend microservices, managing authentication, rate limiting, and SSL termination.
*   **IP Address**: The unique identifier of a machine on a network.
*   **Port**: A numerical identifier (0-65535) that routes traffic to a specific application running on a machine.
*   **Latency**: The time it takes for a request to travel from client to server and back.
*   **Throughput**: The volume of requests a system can process per second.
